Added new `RQ3` conditions in `configs/experiments.yaml`:

  - `A2_history_to_B`
  - `A3_history_to_B`
- summary track
  - `A2_summary_to_B`
  - `A3_summary_to_B`

Interpretation:

- `A_history_to_B` or `A_summary_to_B`
  - one warm-up block in persona `A`
- `A2_*`
  - two warm-up blocks in persona `A`
- `A3_*`
  - three warm-up blocks in persona `A`

The original simple-switch conditions from `RQ1` are intentionally reused:

- `A_history_to_B`
- `A_summary_to_B`
- `B_only`

Added:

- `scripts/analyze_rq3_reinforcement.py`

## Important Behavioral Notes

### Resume behavior

The current resume key does not include `run_id`.

If the same:

- model
- condition
- persona pair
- sample ID
- trial ID

is rerun later, the pipeline may skip it as already completed.

For clean reruns, it is safer to use:

```powershell
--no-resume
```

### Raw-status filtering

Both new analysis layers (`RQ2` and `RQ3`) explicitly filter against raw generation success status.

This prevents failed provider calls or empty generations from silently contaminating downstream summary tables.
